# Telegram Bot UI Patterns

Patterns for building consistent inline-keyboard menus, callbacks, and paginated help messages in this project.

---

## 1. Callback Data Pattern

All inline button callbacks follow a fixed positional schema:

```
<cb_key> <identifier> <action> [<param1>] [<param2>]
```

| Position    | Role                                              | Example value      |
|-------------|---------------------------------------------------|--------------------|
| `cb_key`    | Module prefix — routes to the right handler      | `userset`, `help`, `mrkt`, `wallets` |
| `identifier`| Scopes the action to an owner (user_id, bot_id)  | `5263564829`       |
| `action`    | What to do                                        | `reset`, `modify`, `preview`, `back`, `close`, `page` |
| `param1`    | Subject of the action (setting key, section, …)  | `upload_as_doc`    |
| `param2`    | Secondary param (new value, page number, …)       | `t` / `f`, `0`    |

**Parsing convention** — always unpack with a safe default tail:
```python
data = query.data.split()
_, d1, d2, d3, d4, d5, *_ = data + [None] * 6
```

### Button position constants — ALWAYS use `ButtonPositions`

```python
from bot.utils.constants import ButtonPositions

# ✅ correct
buttons.callback_button(label, callback, position=ButtonPositions.HEADER)
buttons.callback_button(label, callback, position=ButtonPositions.FOOTER)
buttons.add_navigation_buttons(cb_key, identity, position=ButtonPositions.FOOTER)

# ❌ never hardcode strings
buttons.callback_button(label, callback, position="header")   # forbidden
buttons.callback_button(label, callback, position="footer")   # forbidden
```

| Constant | Value | Use for |
|----------|-------|---------|
| `ButtonPositions.HEADER` | `"header"` | Filter rows, action buttons above body |
| `ButtonPositions.FOOTER` | `"footer"` | Navigation, Back/Close rows below body |
| `ButtonPositions.BODY` | `None` | Default (body grid) — omit the kwarg entirely |

> **Import**: `from bot.utils.constants import ButtonPositions` — already re-exported from `bot/utils/constants/__init__.py`.

```python
from bot.utils.constants import ButtonText
```

### Standardized inline button styles

Always pass `style=` explicitly when adding Cancel or Close inline buttons:

```python
# Cancel inline button — SUCCESS style
buttons.callback_button(ButtonText.CANCEL, callback_data, style=ButtonStyle.SUCCESS)

# Close inline button — DANGER style
buttons.callback_button(ButtonText.CLOSE, callback_data, style=ButtonStyle.DANGER)
```

| Button | Style | Notes |
|--------|-------|-------|
| `ButtonText.CANCEL` (inline) | `ButtonStyle.SUCCESS` | Used in callback menus (search, file-select, ytdlp, etc.) |
| `ButtonText.CLOSE` (inline) | `ButtonStyle.DANGER` | Used to dismiss/delete inline menus |
| `ButtonText.CANCEL` (keyboard) | n/a | `add_keyboard_button(ButtonText.CANCEL)` — reply-keyboard wizard flows, no style |

> `add_navigation_buttons` already applies `DANGER` to its Close button internally — no extra `style=` needed there.

## 4. Help Pagination Pattern (`bot/modules/help.py`)

### Structure

Each command's help data (`COMMAND_USAGE[key]`) is stored as:

```python
{
    "main":         ["page0_text", "page1_text", ...],  # split at line boundaries, ≤4096 chars each
    "sections":     {"SectionName": ["page0", "page1", ...], ...},
    "section_names": ["SectionName", ...],              # ordered for button layout
}
```

### Public helpers

```python
# Build COMMAND_USAGE for all help dicts (called once at bot startup)
create_help_buttons()

# Get text + buttons for the initial help display (page 0 of main)
text, markup = get_help_menu(message, key)
```

### Internal helpers

```python
_split_text(text, max_len=4096)  -> list[str]
_make_main_buttons(command_key, section_names, page, total_pages)   -> InlineKeyboardMarkup
_make_section_buttons(command_key, section, page, total_pages)      -> InlineKeyboardMarkup
```

### Callback routing in `help_menu_callback`

| Callback data              | Action                                       |
|----------------------------|----------------------------------------------|
| `help noop`                | Answer only (page counter display button)    |
| `help close`               | Delete message                               |
| `help back <key>`          | Return to main page 0 of command             |
| `help <key> page <n>`      | Navigate to main-text page `n`               |
| `help <key> <section> <n>` | Navigate to section page `n`                 |

### Adding a new help dict

1. Add your `MYMODULE_HELP_DICT` to `bot/utils/help_messages.py` with a `"main"` key and any section keys.
2. Import it in `bot/modules/help.py`.
3. Call `_build_command_usage(MYMODULE_HELP_DICT, "my_key")` inside `create_help_buttons()`.
4. Long sections are paginated automatically — no extra work needed.

## 5. Naming Conventions

- `cb_key`: lowercase, underscore or no separator (`userset`, `mrkt`, `help`, `wallets`)
- `action`: short lowercase verb (`reset`, `modify`, `back`, `close`, `page`, `tog`, `set`)
- Page numbers: **0-indexed integers** as strings in callback data
- Always guard against `None` when parsing positional params:
  ```python
  page = int(d3) if d3 and d3.isdigit() else 0
  ```

## 6. Callback Data Length Limit

Telegram enforces a hard **64-byte limit** on `callback_data`. Every character in the string counts, including spaces.

**Rule of thumb — count before you commit:**

```
bot_manage {user_id} {bot_id} manage
# "bot_manage " (11) + 10-digit user_id + " " + 10-digit bot_id + " manage" (7) ≈ 39 chars  ✅
```

**When combined params risk exceeding the limit, prefer the shorter `bot_manage` prefix instead of `botset`:**

| Prefix     | Characters | Remaining budget (64 - prefix - 1 space) |
|------------|-----------|------------------------------------------|
| `botset`   | 6         | 57                                       |
| `bot_manage` | 10      | 53                                       |

Even though `bot_manage` is longer, it separates user_id and bot_id as positional fields (d1 and d2), keeping each subsequent param short. Use `botset` for single-owner, single-resource actions; use `bot_manage` for anything that carries both user_id *and* bot_id.

**Practical checklist before adding a new callback button:**

1. Write the full callback string with realistic max-length IDs (10 digits each).
2. Count the bytes: `len(callback_str.encode())`.
3. If ≥ 64 bytes, shorten action names, abbreviate params, or restructure the positional schema.

### bot_manage callback schema (multi-identity)

```
bot_manage <user_id> <bot_id_or_action> [<action>] [<param>] [<value>]
```

Root-level (no bot selected):
```
bot_manage {user_id} add                          # open add-bot flow
bot_manage {user_id} allbots                      # dev-only: show all registered bots
bot_manage {user_id} ps {new_start} allbots       # dev-only: paginate allbots view
bot_manage {user_id} back                         # back to user's own bot list
bot_manage {user_id} close                        # close menu
```

Per-bot level (`d2` is a numeric bot_id):
```
bot_manage {user_id} {bot_id} manage              # bot config overview
bot_manage {user_id} {bot_id} plugins             # plugin management
bot_manage {user_id} {bot_id} crud {key}          # field action menu
bot_manage {user_id} {bot_id} ps {new_start} {action}  # paginate manage/plugins
bot_manage {user_id} {bot_id} start|stop|restart  # lifecycle controls
bot_manage {user_id} {bot_id} remove              # remove bot
bot_manage {user_id} {bot_id} toggle_plugin {name}
bot_manage {user_id} {bot_id} default|toggle {key} [value]
bot_manage {user_id} {bot_id} backup|restore manage
bot_manage {user_id} {bot_id} back [action]       # navigation
bot_manage {user_id} {bot_id} close               # close menu
```

Developer-only actions (`user_id == client.dev_id`):
- `allbots` action: displays **all** registered bots from `bot_manager.get_all_clients()` with owner info and pagination.
- Ownership check is bypassed for devs in both `bot_manage_menu` and `bot_manage_callback`, allowing devs to inspect and manage any user's bot.

## `add_force_sub_buttons` — Known Hazards

### Invite link resolution order

`get_chat()` does **not** return `invite_link` for public channels — only `username`. Always
resolve in this order before rendering a ⚠️ fallback button:

```python
invite_link = chat.invite_link
if not invite_link and getattr(chat, "username", None):
    invite_link = f"https://t.me/{chat.username}"
```

### State client must be `client`, not `export_client`

For `default_force_sub_chats`, `export_client = main_client` (a different PyroClient). Callbacks
are handled by the **sub-bot** (`client`). State written to `export_client.state` is stored in
`main_client`'s in-memory dict and is **never reachable** from `client.state.get_user_state()`.

```python
# ✅ Always write to the bot that will receive the callback
await client.state.set_user_state(user_id, {"fsc": "..."})

# ❌ Wrong — main_client.state ≠ client.state; callback handler reads client.state
await export_client.state.set_user_state(user_id, {"fsc": "..."})
```

### Set state in ALL ⚠️ button paths

`global_callback` does `user_state.get("fsc", "No Message")`. If state was never set for the
`fsc` key, the alert always shows **"No Message"**. Call `set_user_state` in every branch that
renders the ⚠️ callback button — including the `not invite_link` path where no exception was
raised.
