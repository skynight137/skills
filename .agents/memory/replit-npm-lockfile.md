## Current state (as of 2026-07-06)

Replit's environment no longer proxies npm through an internal registry — `package-lock.json` generated inside Replit now resolves packages against `registry.npmjs.org` directly. `package-lock.json` is committed for `frontend/` and `xproxy/`, and CI uses `npm ci` with `actions/setup-node`'s `cache: npm`.

**npm version — keep local in sync with CI (2026-07-06):** Node 24 ships with npm 10.9.2 in CI. npm 10 and npm 11 produce different lockfiles; npm 10's `npm ci` fails on lockfiles generated by npm 11 (missing optional transitive entries like `@emnapi/core`/`@emnapi/runtime` from rollup). Fix applied: downgraded local npm to 10.9.2 (`npm install -g npm@10.9.2`), then deleted `node_modules` + lockfile and ran `npm install --include=optional` to regenerate. CI workflow unchanged. **If Replit upgrades its bundled npm or if you reinstall Node, re-pin local npm to whatever version CI's Node 24 ships with.**

If you see `package-firewall.replit.local` (or any non-public host) in a freshly generated lockfile's `resolved` fields, the old constraint has resurfaced — regenerate and diff before assuming `npm ci` is safe, and consider reverting to `npm install` + gitignoring the lockfile again until resolved.

## Historical context (why this rule used to exist)

Replit's npm proxy used to rewrite the `resolved` field of every package entry in `package-lock.json`, e.g. `"resolved": "http://package-firewall.replit.local/vite/-/vite-5.4.11.tgz"`. `npm ci` enforces the exact lockfile including `resolved`, so it failed in GitHub Actions (that host is Replit-internal only). `npm install` worked around it by re-resolving against the public registry, at the cost of losing lockfile determinism and dependency caching in CI.
