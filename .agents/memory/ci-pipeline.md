## What exists

`.github/workflows/ci.yml` — triggers on push and pull_request to `dev`.

### Jobs

| Job | Runner | Key steps |
|-----|--------|-----------|
| `test` | ubuntu-latest | `astral-sh/setup-uv@v6` (Python 3.14 + uv cache); `uv sync --frozen --extra test`; `uv run pytest --tb=short -q`; MongoDB 7 service container on port 27017 |
| `build-frontend` | ubuntu-latest | `actions/setup-node@v4` (Node 24); `npm ci` in `frontend/`; `npm run build` |

Both jobs run in parallel (no dependency between them).

### release-please gating

`.github/workflows/release-please.yml` uses `workflow_run` to fire only after a successful `CI` run on `dev`. This prevents a release PR from being created or merged when tests are red.

## npm version drift — known gotcha

`actions/setup-node@v4` with `node-version: "24"` installs whatever Node 24.x.y the runner cache has, which ships a different npm version than what's locally installed. This has caused two classes of failure:

1. **Lock file out of sync** — if the runner's npm resolves a transitive dep to a newer version than what's in `package-lock.json`, `npm ci` rejects it (`Missing: @emnapi/core@1.11.1`). Fix: regenerate `package-lock.json` with the same npm version CI uses (check setup-node log for `npm: X.Y.Z`).
2. **npm internal crash** — `Exit handler never called!` — npm process killed mid-install (OOM on runner). Fix: add `cache: "npm"` + `cache-dependency-path` to setup-node and pass `--prefer-offline` to `npm ci` so packages come from cache rather than fresh network downloads.

**How to apply:** When changing frontend deps, always regenerate `package-lock.json` by running `npm install` locally with the same npm version CI uses (check the setup-node log from the last green CI run). Never commit a lock file generated by a different npm major/minor.

**Why:** CI runner Node 24.17.0 ships npm 11.13.0; local dev on Node 24.13.0 ships npm 11.6.2. These two resolve the same `^1.x` transitive dep ranges to different concrete versions, making `npm ci` reject the stale lock.

**How to apply:** When adding new test dependencies, add them under `[project.optional-dependencies] test` in `pyproject.toml`; the CI step uses `--extra test` so they are automatically included. Do not add test deps to the main dep list.
