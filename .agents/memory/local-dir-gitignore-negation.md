---
name: .local/ platform-managed dirs vs gitignore negation
description: Why .local/skills and .local/secondary_skills kept vanishing from git tracking, and how to selectively un-ignore a subtree under a global blanket-ignore.
---

## The symptom
`.local/skills/` and `.local/secondary_skills/` repeatedly appeared untracked/deleted from git's
perspective across sessions, while `.local/custom_skills/` (meant to hold user-authored content)
seemed to disappear too.

## Root cause
`.local/skills/` and `.local/secondary_skills/` are platform-managed, regenerated-per-session
directories (confirmed via `.local/skills/skill-authoring/SKILL.md` stating `.local/skills/` is
read-only/platform-managed, plus matching `.fingerprint` files/mtimes). They are not meant to be
tracked at all. Only `.local/custom_skills/` is genuinely user-owned and worth tracking.

Separately, Replit's global `/etc/.gitignore` (wired in via `core.excludesFile`) blanket-ignores
`.local/` for every repo. A repo's own `.gitignore` can override a global excludesFile pattern via
negation (`!pattern`), but negation has a recursion rule: git will not look inside a directory it
has already decided to ignore, so un-ignoring a directory and then trying to re-ignore/un-ignore
files *inside* it requires un-ignoring each level of the path, not just the leaf.

## The fix
In the project's own `.gitignore`, add a block (order matters — later patterns override earlier
ones of the same specificity, but you still need to un-ignore each ancestor directory):
```
!.local/
.local/bin/
.local/share/
.local/state/
.local/skills/
.local/secondary_skills/
!.local/custom_skills/
!.local/custom_skills/**
```
`!.local/` un-ignores the directory itself so git will traverse into it at all; the platform-owned
subdirs are then explicitly re-ignored; `.local/custom_skills/` is un-ignored again, plus `/**` to
also un-ignore everything inside it (a bare `!dir/` does not automatically un-ignore its
contents).

**Why:** matches the intended ownership boundary — `.local/skills/` and `.local/secondary_skills/`
are regenerated by the platform every session and would just create endless spurious diffs if
tracked; `.local/custom_skills/` is the only subtree actually authored by the user/agent.

**How to apply:** if `.local/skills` or `.local/secondary_skills` show up as modified/deleted in
`git status` again, check this `.gitignore` block hasn't been overwritten or reordered — do not
try to "fix" it by tracking those dirs, since they'll just get regenerated differently next
session.

## Verification method that actually works
`git check-ignore -v <path>` prints the matching pattern for *any* path checked against the
ignore rules, including negation patterns — so a non-empty result does not necessarily mean the
path is ignored. Use `git check-ignore -q <path>; echo $?` (0 = ignored, 1 = not ignored) per
path, and confirm with an actual `git add` test on a marker file, rather than trusting `-v`
output at a glance.
